## 一、产品概述

### 1.1 产品定位

**一句话定义**：一个本地运行的 Web 应用，将 Obsidian Vault 中的 Markdown 笔记可视化为交互式知识图谱，并结合 AI 对用户的知识储备进行智能分析。

### 1.2 目标用户

| 用户画像 | 特征 |
|---------|------|
| **主要用户**：你（hanyutian） | 技术专业人士，关注 AI Agent、自动化工具、开发效率 |
| **使用场景** | 个人知识管理复盘、发现知识盲区、梳理知识体系 |
| **现有资产** | Obsidian Vault `~/Documents/Obsidian Vault/`，35 篇 Markdown 笔记 |

### 1.3 核心价值主张

> **从"零散的 Markdown 文件"到"可视化的知识体系 + AI 洞察"**

- **看得见**：知识图谱可视化，一眼看清笔记之间的关联
- **看得懂**：AI 自动聚类、总结、发现知识主题
- **看得远**：识别知识缺口、追踪关注趋势、给出学习建议

```
Obsidian Vault/
└── 第二大脑/
    ├── claude_agent/       # 5 个 agent 配置文件（code-reviewer, developer, ppt-maker, vision-reader, agent清单）
    ├── skills/             # 3 个 skill 说明文档
    ├── codex/              # Codex 配置教程
    └── 公司ai应用/          # AI 业务应用实践文档
```

| 维度 | 现状 |
|------|------|
| 笔记总数 | **35 篇** Markdown |
| 标签体系 | `#agent`, `#claude`, `#skill`, `#deepseek`, `#codex`, `#workbuddy`, `#视频生成`, `#剪映` 等 |
| Wiki 链接 | **7 个**双向链接 |

| 知识领域 | AI Agent、自动化工具、开发效率、PPT 制作、视频生成 |

### 2.2 当前痛点

1. **关联不可见**：35 篇笔记之间的引用关系仅靠 7 个 Wiki 链接，大量隐含关联未被显式化
2. **标签利用率低**：虽然打了标签，但难以从标签维度快速了解知识分布
3. **缺乏全局视角**：不知道自己的知识体系中哪些领域强、哪些领域弱
4. **时间维度缺失**：无法追踪最近关注什么、知识增长趋势
5. **AI 未介入**：虽然有大量 AI 相关笔记，但 AI 从未分析过笔记内容本身

#### F1：Vault 数据解析引擎（P0 - 必须）

**功能描述**：读取指定 Obsidian Vault 目录，解析所有 Markdown 文件的结构化信息。

| 解析内容 | 提取方式 | 示例 |
|---------|---------|------|
| 文件名 + 路径 | 文件系统读取 | `claude_agent/developer.md` |
| YAML Frontmatter | 正则 / gray-matter 解析 | `tags: [agent, claude, 开发]` |
| Wiki 链接 `[[...]]` | 正则提取 | `[[code-reviewer]]` |
| 标签 `#tag` | 正则提取 | `#agent`, `#skill` |
| 标题层级 | Markdown AST 解析 | `## 你的主要职责` |
| 外链 URL | 正则提取 | `https://api.deepseek.com` |
| 文件修改时间 | `fs.stat` | `2026-05-28` |
| 内容摘要 | 取前 200 字正文 | — |

**输入**：Vault 根目录路径（可配置）

**输出**：
```json
{
  "nodes": [
    {
      "id": "claude_agent/developer.md",
      "title": "developer",
      "tags": ["agent", "claude", "开发"],
      "links": ["code-reviewer"],
      "mtime": "2026-05-28T10:00:00Z",
      "summary": "日常开发主力 agent。编写代码...",
      "domain": "claude_agent",
      "centrality": 0.42
    }
  ],
  "edges": [
    { "source": "claude_agent/agent清单.md", "target": "claude_agent/developer.md", "type": "wiki-link" },
    { "source": "claude_agent/developer.md", "target": "claude_agent/code-reviewer.md", "type": "tag-co-occurrence" }
  ]
}
```

**边（关系）类型**：
- `wiki-link`：显式 `[[双向链接]]`
- `tag-co-occurrence`：共享 ≥ 2 个相同标签
- `content-similarity`：AI 语义相似度（可选，见 F3）
- `same-directory`：同目录归属

---

#### F2：知识图谱可视化（P0 - 必须）

**功能描述**：以力导向图（force-directed graph）展示笔记节点及其关联。

**技术要求**：
- 使用 **D3.js force simulation** 或 **vis-network**
- 支持 50-500 节点规模流畅渲染
- Canvas/WebGL 渲染（非 SVG，保证性能）

**视觉设计**：

| 视觉元素 | 规则 |
|---------|------|
| 节点大小 | 与「被引用次数 + 标签数」正相关 |
| 节点颜色 | 按知识领域（目录）区分色系 |
| 边粗细 | 与关联强度（共现标签数）正相关 |
| 边颜色 | 按关系类型区分（实线=wiki-link，虚线=tag，点线=similarity） |

**交互行为**：
- **悬停**：显示笔记标题 + 摘要 tooltip
- **单击**：高亮该节点及其一度邻居，右侧显示详情面板
- **双击**：用默认编辑器打开原 Markdown 文件
- **拖拽**：拖拽节点调整布局
- **滚轮**：缩放画布
- **框选**：按住 Shift 拖拽框选多节点

**布局模式**：
1. **力导向布局**（默认）：自动排布
2. **按目录分组**：节点按所属文件夹聚集
3. **按时间排列**：X 轴为时间线，Y 轴为目录

#### F3：AI 知识分析引擎（P1 - 重要）

**功能描述**：调用 LLM 对笔记内容进行语义分析，产出洞察报告。

##### F3.1 知识聚类分析

- 输入：所有笔记的标题 + 摘要 + 标签
- AI 任务：将笔记自动分为 3-7 个主题簇（cluster），为每个簇命名
- 输出：聚类标签 + 每个簇的笔记列表
- 示例输出：
  ```
  簇 1「AI Agent 工具链」(12篇)：code-reviewer, developer, ppt-maker...
  簇 2「视频内容生产」(5篇)：viral-video-producer, codex自动生成视频工程...
  簇 3「工作方法论」(8篇)：本地AI Agent业务应用实践...
  ```

##### F3.2 知识缺口识别

- 输入：知识聚类结果 + 用户自定义的兴趣领域
- AI 任务：对比当前知识覆盖与推荐知识体系，识别缺失主题
- 输出：缺口列表 + 每个缺口的学习建议
- 示例输出：
  ```
  🔴 缺口「Prompt Engineering 基础」：你的笔记聚焦 Agent 应用，但缺少 Prompt 设计原理
  🟡 缺口「模型评测与选型」：未涉及不同模型的能力对比和选型决策框架
  ```

##### F3.3 时间趋势分析

- 输入：笔记修改时间 + 标签
- AI 任务：分析最近 7/30/90 天的关注热点变化
- 输出：趋势描述 + 时间线可视化
- 示例输出：
  ```
  📈 近 30 天你关注最多的是：Agent 配置（5篇新增/修改）
  📉 关注度下降：视频制作（30天内无更新）
  🆕 新兴关注：DeepSeek 集成（最近 7 天新增）
  ```

##### F3.4 知识体系总结

- AI 任务：基于全部笔记，用 3-5 句话概括用户的知识体系
- 输出：一段自然语言总结
- 示例：
  > 你的知识体系以「AI Agent 工程化落地」为核心，涵盖 agent 设计模式、PPT/视频内容自动化生产、以及企业级 AI 应用实践。你的强项在于工具链集成和实操落地，在理论基础和社区贡献方面还有拓展空间。

#### F4：智能搜索与过滤（P1）

| 功能 | 描述 |
|------|------|
| **关键词搜索** | 模糊匹配文件名、标签、正文内容，高亮匹配节点 |
| **标签过滤** | 多选标签，图谱仅显示包含任一选中标签的节点 |
| **时间范围过滤** | 滑块选择时间窗口，仅显示该时间段内修改过的笔记 |
| **领域过滤** | 按目录/知识领域筛选 |
| **路径查找** | 选中两个节点，高亮它们之间的最短连接路径 |

#### F5：详情面板（P2）

单击节点后，右侧面板显示：

## 五、技术方案

### 5.1 架构设计

```
┌──────────────────────────────────────────────────┐
│                   浏览器前端                        │
│  ┌──────────┐ ┌──────────┐ ┌──────────────────┐  │
│  │ 知识图谱  │ │ AI 面板   │ │  搜索/过滤/设置   │  │
│  │ D3.js    │ │ Markdown │ │                  │  │
│  └──────────┘ └──────────┘ └──────────────────┘  │
├──────────────────────────────────────────────────┤
│               本地 API Server (Python/FastAPI)     │
│  ┌──────────┐ ┌──────────┐ ┌──────────────────┐  │
│  │ Vault    │ │ AI       │ │  Graph Data API  │  │
│  │ Parser   │ │ Analysis │ │                  │  │
│  └──────────┘ └──────────┘ └──────────────────┘  │
├──────────────────────────────────────────────────┤
│               Obsidian Vault (文件系统)             │
└──────────────────────────────────────────────────┘
```

### 5.2 技术选型

| 层 | 技术 | 理由 |
|----|------|------|
| **后端** | Python + FastAPI | 你的环境已有 Python 3.14；FastAPI 轻量、自动生成 API 文档 |
| **Markdown 解析** | `python-frontmatter` + `mistune` | 提取 frontmatter 和结构化内容 |
| **图数据模型** | `networkx` | Python 端构建图结构，计算中心性等指标，导出 JSON 给前端 |
| **前端框架** | **纯 HTML/CSS/JS**（推荐）或 Vue 3 | 优先轻量，单文件即可运行 |
| **图谱渲染** | **D3.js v7**（force simulation）+ Canvas | 行业标准，灵活可控，支持 500+ 节点 |
| **AI 调用** | Anthropic SDK / OpenAI SDK / DeepSeek API | 根据你已有的 API Key 选择；建议先用 DeepSeek（成本低） |
| **本地文件访问** | FastAPI `StaticFiles` + 文件系统读取 | 通过本地 HTTP API 读取 vault 文件，不需要 Electron |
| **UI 样式** | Tailwind CSS（CDN） | 快速构建美观界面 |

### 5.3 为什么不选 Electron？

| 方案 | 优点 | 缺点 |
|------|------|------|
| **Web + 本地 Server**（推荐） | 轻量、开发快、无需打包 | 需要手动启动 server |
| Electron 桌面应用 | 一体化体验、原生文件访问 | 开发成本高、打包体积大（150MB+） |
| 纯前端 File API | 零依赖 | 每次需手动选择目录，无法持久化配置 |

> **建议**：MVP 阶段使用「Web + 本地 Server」，后续可升级为 Electron 或 Tauri 打包。

### 5.4 API 设计（草案）

```
GET  /api/vault/scan?path=<vault_path>        → 全量扫描 vault，返回图数据
POST /api/vault/refresh                        → 增量刷新（检测文件变更）
GET  /api/file/content?path=<rel_path>         → 获取单文件完整内容
POST /api/analysis/cluster                     → AI 聚类分析
POST /api/analysis/gaps                        → AI 知识缺口分析
POST /api/analysis/trend?days=30               → AI 趋势分析
POST /api/analysis/summary                     → AI 知识体系总结
GET  /api/search?q=<keyword>&tags=<tags>       → 关键词/标签搜索
```

## 八、风险与应对

| 风险 | 影响 | 概率 | 应对 |
|------|------|------|------|
| Vault 规模增长后图谱性能下降 | 用户体验差 | 中 | 使用 Canvas 渲染 + 虚拟化；节点数 > 200 时自动聚合 |
| AI 分析结果不稳定（LLM 幻觉） | 分析不可信 | 中 | 结构化 prompt + 温度设低 + 人工可纠错 |
| 用户长期不使用 | 产品价值未验证 | 高 | MVP 快速上线验证；添加定期提醒 |
| 文件编码问题（中文路径/内容） | 解析失败 | 低 | 统一 UTF-8 + 异常捕获 |
| 安全：本地文件暴露 | 数据泄露 | 低 | Server 仅监听 127.0.0.1，不暴露到网络 |

## 九、竞品分析

| 产品 | 优势 | 劣势 |
|------|------|------|
| **Obsidian 内置图谱** | 原生集成、双向链接 | 无 AI 分析、不可定制、不支持语义聚类 |
| **Logseq** | 开源、大纲式 | 图谱功能较弱、无 AI |
| **Roam Research** | 块引用强大 | 需联网、价格高 |
| **Notion AI** | AI 集成好 | 非本地、非图谱视角、隐私问题 |
| **本产品** | 本地优先 + 知识图谱 + AI 分析三位一体 | MVP 功能少，需自行部署 |
